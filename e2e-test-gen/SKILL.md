# E2E Test Case Generator

根据需求文档生成结构化 E2E 测试用例，输出为 Markdown 文档。

## ⚠️ 重要：默认按实验组纬度组织

**当需求文档中包含实验组（如对照组、实验组1、实验组2等）时，必须按实验组纬度组织测试用例！**

实验组纬度结构：
```
# 一、对照组
## 1.1 功能模块A
## 1.2 功能模块B

# 二、实验组1
## 2.1 功能模块A
## 2.2 功能模块B
...
```

每个实验组内的测试点需标注该组的特定预期结果。

## Workflow

```
需求文档 → 功能模块分析 → 提取测试点 → 用户确认 → 生成 Markdown 文档
```

### Step 1: 接收需求文档

获取需求文档内容。支持的输入方式：
- 直接粘贴需求文本
- 提供文档路径/URL（读取后分析）
- 提供飞书文档链接（使用 feishu_fetch_doc 获取）

### Step 2: 分析需求，提取功能模块与测试点

阅读 [references/test-patterns.md](references/test-patterns.md) 获取测试设计模式参考。

对需求文档进行系统分析：
1. 识别功能模块（每个主要功能区域作为一个模块）
2. 每个模块下提取测试点，覆盖：
   - 正常流程（Happy Path）
   - 异常流程（错误处理、边界条件）
   - 数据场景（空数据、大量数据、特殊字符）
   - 权限场景（不同角色、越权访问）
   - 关联场景（模块间交互）

### Step 3: 展示功能点清单，等待用户确认

以结构化列表形式展示，格式：

```
📋 模块一：[模块名称]
  1. [测试点1]
  2. [测试点2]
  3. [测试点3]

📋 模块二：[模块名称]
  1. [测试点1]
  2. [测试点2]

...
```

向用户确认：
- 是否需要增减测试点
- 是否有遗漏的业务场景
- 确认后进入生成阶段

### Step 4: 生成完整测试用例

确认后，为每个测试点补充完整信息。

**输出格式**：默认生成 Markdown 文档格式

#### 测试用例单行格式（必须）

每个测试用例的格式为：

```markdown
TC-001：测试点名称 | 前置条件：... | 测试步骤：1. xxx 2. xxx | 预期结果：...
```

**重要**：前置条件、测试步骤、预期结果**必须放一行**，用" | "分隔。测试步骤中的多步骤用空格分隔（1. xxx 2. xxx）。

#### 4.1 按实验组纬度生成（默认）

当需求包含实验组时，使用实验组纬度结构：

```markdown
# 一、对照组

## 1.1 功能模块

TC-001：测试点名称 | 前置条件：... | 测试步骤：1. xxx 2. xxx | 预期结果：...

TC-002：测试点名称 | 前置条件：... | 测试步骤：1. xxx | 预期结果：...

## 1.2 另一个模块

...

# 二、实验组1

## 2.1 功能模块

TC-xxx：测试点名称 | 前置条件：... | 测试步骤：1. xxx | 预期结果：该实验组的特定预期结果
```

#### 4.2 按功能模块纬度生成

当需求不包含实验组时，使用功能模块结构：

```markdown
# 一、功能模块A

TC-001：测试点 | 前置条件：... | 测试步骤：1. xxx 2. xxx | 预期结果：...
```

### Step 5: 输出文档

将生成的 Markdown 内容：
1. 保存到 `/tmp/openclaw/` 目录（文件名以 .md 结尾）
2. 根据用户需求选择输出方式：
   - 用户要求"发文件"或"发送文件"：直接发送 md 文件（使用 message 工具发送 path）
   - 用户要求"飞书文档"：使用 feishu_create_doc 创建飞书文档

**重要**：不再默认生成 XMind 文件，如用户明确要求再生成。

## Test Case ID 规则

默认按模块内顺序编号 TC-001, TC-002, ...。若用户要求全局编号则全局递增。

## 质量标准

- 每个测试步骤必须具体、可执行（不能写"操作成功"，要写具体点了什么、输入了什么）
- 预期结果必须可验证（明确的页面跳转、提示信息、数据变化）
- 前置条件不能省略（系统状态、数据准备、登录状态等）
- 覆盖正向 + 异常 + 边界场景

### 实验组纬度测试用例特殊要求

- **必须明确标注每个实验组的特定预期结果**
- **对照组与实验组的差异点需重点覆盖**
- **实验组独有的功能点需单独列出**（如实验组2的弱引导、实验组3的首坑强引导等）
- **共用测试点需说明在哪些实验组执行**
- **前置条件、测试步骤、预期结果放一行**

### 4.3 脑图格式（树形层级结构）

当用户要求脑图格式或树形展示时，使用以下结构：

```markdown
## 一、功能模块

### 1.1 测试功能点

- **TC-001：测试点名称**
  - 前置条件
    - 条件1
    - 条件2
  - 测试步骤
    1. 步骤1
    2. 步骤2
    3. 步骤3
  - 预期结果
    - 结果1
    - 结果2

- **TC-002：测试点名称**
  - 前置条件
    - 条件1
  - 测试步骤
    1. 步骤1
  - 预期结果
    - 结果1
```

**关键要求**：
- 使用 `- **TC-xxx**` 表示测试用例
- 使用 `- 前置条件` / `- 测试步骤` / `- 预期结果` 作为二级子项
- 测试步骤中的每一步使用 `1. 步骤内容` 编号
- 预期结果中的每一条使用 `- 结果内容`

## 示例输出

### 单行格式示例

```markdown
TC-001：从商品库选品创建测试商品 | 前置条件：商家已登录后台，商品库中有可用商品 | 测试步骤：1. 进入C端测试商品管理页面 2. 点击"创建测试商品" 3. 选择"从商品库选品" 4. 勾选商品（1-10款）5. 点击确认 | 预期结果：成功创建测试商品，商品显示在测试商品列表中
```

### 脑图格式示例

```markdown
## 一、C端测试商品管理

### 1.1 测试商品创建

- **TC-001：从商品库选品创建测试商品**
  - 前置条件
    - 商家已登录后台
    - 商品库中有可用商品
  - 测试步骤
    1. 进入C端测试商品管理页面
    2. 点击"创建测试商品"
    3. 选择"从商品库选品"
    4. 勾选商品（1-10款）
    5. 点击确认
  - 预期结果
    - 成功创建测试商品
    - 商品显示在测试商品列表中
```
